# FluxGate UI Clarity Design

## 背景

FluxGate 是基于 v2rayN 7.10.3 的 Windows 代理客户端二次开发项目。当前阶段的核心目标不是重写代理内核，而是把 FluxGate 的主界面、功能入口和用户路径整理清楚，让普通用户打开软件后能快速判断“现在能不能用、正在用哪个节点、流量怎么走、代理端口是多少、遇到问题该点哪里”。

当前分支是 `codex/canva-ui-preview`，已有六页式 Avalonia 主界面预览：首页、节点、订阅、路由、设置、日志。现有问题不是功能缺失，而是页面职责、按钮优先级、状态表达和视觉密度还不够稳定，容易呈现为“把 v2rayN 功能重新摆了一遍”。

本设计新增三个确认项：

- TUN 模式必须纳入首页状态、路由页说明和新手指南。
- 首页必须指出本机代理端口，方便用户排障和手动配置其他软件。
- 新增“新手指南/功能介绍”页面，解释基本操作、名词含义，以及何时使用哪种操作。

## 产品定位

FluxGate 的 UI 方向确定为：

> 小白首页作为产品骨架，吸收 Command Center 的视觉气质，但不让控制台式信息密度压住易用性。

这意味着首页必须优先服务普通用户的即时判断和快速操作；科技感、状态卡片、日志、指标和高级入口只能帮助理解当前代理状态，不能反过来占据主流程。

## 第一屏原则

用户打开 FluxGate 后，第一屏只优先回答五个问题：

1. 现在能不能用？
   - 显示已连接、未连接、连接异常、Core 未启动等主状态。
   - 主状态必须比装饰性指标更醒目。

2. 我正在用哪个节点？
   - 显示当前节点名称、分组、延迟或最近状态。
   - 提供进入节点页或快速切换节点的入口。

3. 流量怎么走？
   - 显示系统代理状态、路由模式和 TUN 模式状态。
   - 用普通用户能理解的表达，例如自动、全局、直连、规则、TUN 已开启/未开启，而不是优先暴露复杂术语。

4. 代理端口是多少？
   - 首页必须显示本机监听地址和常用端口，例如 `127.0.0.1:10808`。
   - 优先展示用户最常用的 Mixed/HTTP/SOCKS 端口，具体以现有配置为准。
   - 提供复制端口或复制代理地址的按钮，方便配置浏览器、命令行工具或其他软件。

5. 不懂该点哪里时怎么办？
   - 首页提供进入“新手指南”的入口。
   - 新手指南解释基础操作、常见名词和使用场景。

## 页面结构

主导航从原六页扩展为八个能力区。推荐结构如下：

1. 首页：连接控制台
   - 放连接状态、当前节点、系统代理状态、路由模式、TUN 模式状态、代理端口、订阅更新入口、异常提示。
   - 代理端口区域显示本机地址、HTTP/SOCKS/Mixed 等关键端口，并提供复制按钮。
   - 不放完整节点表格，不堆高级设置。
   - 提供“新手指南”入口，给第一次使用的人一个明确出口。

2. 节点：节点工作台
   - 复用现有 `ProfilesView`。
   - 整理顶部工具条：导入、添加、刷新订阅、测试延迟、排序、分组。
   - 保留右键菜单和原有批量操作。

3. 订阅：来源维护入口
   - 主界面提供添加/管理订阅、全部更新、通过代理更新、更新当前分组等清楚入口。
   - 订阅 URL、转换、过滤、备注等复杂字段继续进入现有订阅设置弹窗。

4. 免费节点：免费节点中心
   - 在 FluxGate 内部提供 WebView 页面，加载 `https://lovable.dev/preview/lZwTAW5Wyepb3fplbbhlJLUZpa7z6kCO`。
   - 同时保留“在浏览器打开”按钮作为兜底。
   - FluxGate 不保存该网页数据，不注入脚本，不读取网页 Cookie，只作为展示和跳转入口。
   - 该功能需要单独评估 Avalonia WebView 方案、包体积、Windows 运行依赖和发布包影响。

5. 路由：模式与规则入口
   - 主界面用小白可理解的方式展示自动、全局、直连、规则等模式。
   - 明确展示 TUN 模式入口和状态。
   - geosite、geoip、domainStrategy、DNS 等复杂编辑继续进入原有弹窗。

6. 设置：常用设置索引
   - 主界面只展示常用入口，例如高级设置、热键、备份、打开配置目录、清理统计。
   - 不把 `OptionSettingWindow` 的全部参数搬到主界面。
   - 端口修改仍进入现有设置弹窗，首页只负责展示和复制。

7. 日志：排障视图
   - 复用现有 `MsgView`。
   - 让用户知道“出问题先看这里”，并提供打开日志目录等入口。

8. 新手指南：功能介绍与名词解释
   - 解释软件的基本操作路径。
   - 解释节点、订阅、系统代理、代理端口、路由模式、TUN 模式、Core、延迟测试、日志等名词。
   - 用“什么时候用什么操作”的方式组织内容，而不是写成技术文档。

## 首页代理端口设计

首页需要有一个小而清楚的“本机代理端口”区域，建议放在连接状态和当前节点附近，作为排障信息而不是主操作按钮。

展示内容：

- 本机地址：`127.0.0.1`
- Mixed 端口：优先展示，如果现有配置启用。
- HTTP 端口：如果现有配置启用。
- SOCKS 端口：如果现有配置启用。
- 复制按钮：复制完整地址，例如 `127.0.0.1:10808`。
- 设置入口：跳转到现有高级设置弹窗，不在首页直接编辑端口。

展示原则：

- 首页显示“当前正在监听的端口”，不要写死端口数字。
- 如果端口未启用，显示“未启用”或隐藏对应行，避免误导用户。
- 如果 Core 未启动，端口区域应提示“Core 未启动，端口暂不可用”。
- 不展示公网 IP，不展示订阅链接，不展示任何敏感数据。

## 新手指南内容设计

新手指南页面建议采用卡片式结构，每张卡片回答一个实际问题。

### 基本操作

1. 第一次使用
   - 添加订阅或导入节点。
   - 更新订阅。
   - 选择一个节点。
   - 开启系统代理。
   - 打开浏览器测试网页。

2. 换节点
   - 进入节点页。
   - 看延迟或名称选择节点。
   - 双击或使用现有命令设为活动节点。
   - 如果网页打不开，先换节点，再看日志。

3. 更新节点
   - 有订阅时使用“刷新订阅”。
   - 如果更新失败，可以尝试“通过代理更新”。
   - 如果没有订阅，可以进入“免费节点中心”获取节点入口。

4. 手动配置其他软件
   - 先看首页的本机代理端口。
   - 在其他软件里填入 `127.0.0.1` 和对应端口。
   - 如果软件支持系统代理，优先使用系统代理；如果不支持，再手动填写端口。

5. 排查问题
   - 先看首页主状态。
   - 再看当前节点是否可用。
   - 再看系统代理/TUN 是否符合当前场景。
   - 再看代理端口是否已启用。
   - 最后进入日志页查看错误信息。

### 名词解释

1. 节点
   - 可以理解为一条代理线路。
   - 节点质量会影响网页打开速度和稳定性。

2. 订阅
   - 可以理解为一组节点的在线列表。
   - 刷新订阅就是从订阅地址重新获取节点。

3. 系统代理
   - 让浏览器、聊天软件等支持系统代理的程序走 FluxGate。
   - 常规网页浏览优先使用系统代理。

4. 代理端口
   - 可以理解为 FluxGate 在本机开的“入口门牌号”。
   - 其他软件如果不能自动识别系统代理，可以手动填写 `127.0.0.1` 和端口号。
   - 常见端口类型包括 HTTP、SOCKS 和 Mixed，最终以软件首页显示为准。

5. 路由模式
   - 自动/规则：常用推荐模式，国内直连、国外走代理。
   - 全局：所有流量尽量走代理，适合排查或临时需要。
   - 直连：不走代理，适合确认是否是代理导致的问题。

6. TUN 模式
   - 可以理解为更底层的接管方式，让不支持系统代理的软件也能走代理。
   - 适合游戏、命令行工具、部分桌面软件或不遵守系统代理的软件。
   - 不建议新手默认开启；只有“浏览器能用，但某些软件不能用”时再考虑。
   - 开启前应提示可能需要管理员权限，并可能影响网络环境。

7. Core
   - 真正负责代理连接的核心程序。
   - FluxGate 是界面和控制中心，Core 是底层发动机。

8. 延迟测试
   - 用来粗略判断节点响应速度。
   - 延迟低不一定代表网页最快，最终仍以真实浏览器打开网页体验为准。

9. 日志
   - 记录 Core 启动、连接、订阅更新、错误信息。
   - 遇到问题时用于排查。

### 何时用哪种操作

1. 只是上网页
   - 推荐：选择节点，开启系统代理，路由模式用自动/规则。

2. 浏览器能用，但某个软件不能用
   - 推荐：先检查软件是否支持系统代理。
   - 如果支持手动代理，填首页显示的本机代理端口。
   - 如果不支持系统代理和手动代理，再考虑开启 TUN 模式。

3. 所有网页都打不开
   - 推荐：换节点，刷新订阅，看日志。
   - 同时确认首页代理端口是否启用。
   - 不要第一时间乱改高级路由和 DNS。

4. 国外网站打不开，国内网站正常
   - 推荐：检查当前节点、系统代理、路由模式。
   - 可临时切换全局模式排查。

5. 国内网站也变慢
   - 推荐：检查是否误开全局模式或 TUN 模式。
   - 恢复自动/规则模式。

6. 节点越来越少或失效
   - 推荐：刷新订阅。
   - 如果没有订阅，进入免费节点中心。

## TUN 模式界面设计

TUN 模式需要出现三个位置：

1. 首页状态卡片
   - 显示“系统代理：已开启/未开启”。
   - 显示“TUN 模式：已开启/未开启”。
   - 同区域显示代理端口，帮助用户区分“系统代理、手动端口、TUN”三种入口。
   - 如果 TUN 开启，首页要给出清楚提示，避免用户忘记自己正在使用更强的接管模式。

2. 路由页
   - 给出 TUN 模式入口。
   - 用一句话说明适用场景：“用于让不支持系统代理的软件也走代理”。
   - 提供进入高级设置的入口，而不是在主界面暴露全部 TUN 参数。

3. 新手指南页
   - 解释 TUN 是什么。
   - 解释什么时候需要开启。
   - 解释可能带来的影响，例如需要管理员权限、影响部分网络环境。
   - 对比系统代理、手动代理端口和 TUN 模式的使用场景。

## 阶段路线

### 阶段 1：界面结构收束

目标：让主界面职责和文字表达先清楚。

工作内容：

- 首页改为真正的连接控制台。
- 首页加入系统代理、路由模式、TUN 模式、代理端口状态摘要。
- 导航职责重新整理，纳入免费节点和新手指南能力。
- 免费节点入口从单一顶部按钮规划为独立能力。
- 新手指南作为功能介绍与名词解释页面进入规划。
- 修正明显乱码和不稳定文案。
- 调整按钮主次关系，减少首页信息噪音。
- 不新增代理业务逻辑。

验收：

- `dotnet build .\v2rayN\v2rayN.sln -c Debug` 通过。
- 首页、节点、订阅、免费节点、路由、设置、日志、新手指南能正常切换或打开。
- 首页能显示系统代理、TUN 模式和代理端口摘要。
- 现有弹窗仍能打开。
- 刷新订阅、免费节点外部打开、高级菜单、退出等原命令仍能触发。

### 阶段 2：新手指南页面

目标：让普通用户不用查外部教程，也能理解基本操作和常见名词。

工作内容：

- 新增功能介绍页面。
- 按“第一次使用、换节点、更新节点、手动配置其他软件、排查问题”组织基础操作。
- 按“节点、订阅、系统代理、代理端口、路由模式、TUN 模式、Core、延迟测试、日志”组织名词解释。
- 按“只是上网页、某个软件不能用、所有网页打不开、国内网站变慢、节点失效”等场景解释推荐操作。

验收：

- 页面文字通俗，不堆技术术语。
- 代理端口解释清楚，能指导用户配置其他软件。
- TUN 模式解释清楚但不过度鼓励新手默认开启。
- 页面不需要网络也能阅读。
- 不记录任何订阅链接、Token、Cookie 或密钥。

### 阶段 3：组件化整理

目标：降低 `MainWindow.axaml` 继续膨胀的风险。

工作内容：

- 将重复的页头、状态卡片、工具条、入口卡片抽成样式或 Avalonia UserControl。
- 保持现有 ViewModel 命令复用，不急于拆服务层。
- 控制每次修改范围，避免一次性大重构。

验收：

- 构建通过。
- 页面视觉一致。
- 抽出的组件不改变原命令行为。

### 阶段 4：真实状态增强

目标：让首页状态逐步接入真实数据，而不是只展示静态文案。

工作内容：

- 当前节点状态接入真实配置。
- 系统代理状态接入现有状态栏逻辑。
- TUN 模式状态接入现有配置或状态来源。
- 代理端口接入现有配置或状态来源。
- 订阅更新时间和异常提示逐步接入现有消息机制。
- Core 状态和日志摘要只做展示，不重写 Core 控制。

验收：

- 状态显示与实际代理行为一致。
- TUN 开关状态显示准确。
- 代理端口显示准确，复制内容可用。
- 异常状态不会误导用户。
- 不影响节点切换、订阅更新和路由设置。

### 阶段 5：免费节点中心

目标：实现内嵌 WebView 免费节点页，并保留外部浏览器兜底。

工作内容：

- 评估 Avalonia WebView 可用方案。
- 新增免费节点页入口。
- WebView 默认打开指定 lovable.dev 页面。
- 提供“刷新”和“在浏览器打开”按钮。
- WebView 加载失败时显示明确提示和外部打开按钮。

验收：

- 内嵌页可加载或给出可理解失败提示。
- 外部浏览器按钮始终可用。
- 不保存网页 Cookie、Token、订阅链接或其他敏感数据。
- 打包后 Windows 运行依赖明确。

### 阶段 6：发布与人工验收

目标：确认 UI 预览可交付。

工作内容：

- Debug 构建。
- Release 打包。
- 检查包根目录不包含 `v2rayN.exe`。
- 人工运行 FluxGate 验收 UI。

验收：

- 构建和打包成功。
- TUN 模式状态展示符合实际配置。
- 代理端口展示符合实际配置。
- 新手指南内容完整且容易理解。
- 免费节点入口行为符合预期。
- 不与用户本机已有 v2rayN 同时运行，避免争用系统代理、Core 端口和托盘状态。

## 决策记录

- 选择“小白首页 + Command Center 气质”，不选择纯控制台密集布局。
- 保留原有主功能结构，但重新定义每页职责。
- 免费节点从外部跳转升级为“内嵌 WebView + 外部打开按钮”。
- TUN 模式纳入首页状态、路由入口和新手指南解释。
- 代理端口纳入首页状态和新手指南解释，支持复制完整本机代理地址。
- 新增新手指南/功能介绍页面，用于解释基础操作、名词和使用场景。
- 当前阶段只做 UI 结构和入口整理，不碰代理核心。

## 后续实施计划入口

本设计通过后，再单独编写 implementation plan。实施计划应按任务拆分，优先从阶段 1 和新手指南页面开始，避免一次性同时引入 WebView、组件化和真实状态接入。
